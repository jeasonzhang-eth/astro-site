# Jeason × 倍数引擎一体化网站设计规格

## 1. 背景

当前 `astro-site` 是 Jeason Zhang 的双语个人网站，核心内容包括个人定位、真实软件项目、工作笔记和作者页面。服务器上原有的 `beishuyinqing.cn` 则是一个单页公司官网，承载深圳市倍数引擎软件技术有限责任公司的业务介绍、公司信息、联系方式和 ICP 备案信息。

本次改造的目标不是把两个站点机械拼接，而是建立一个统一网站：

- Jeason 的个人身份负责建立专业可信度和清晰的创作者视角。
- 真实项目负责提供可检查的能力证据。
- 倍数引擎负责承接商业服务、合作关系和公司主体信息。
- `beishuyinqing.cn` 最终由整合后的 Astro 站点提供服务，并保留可快速回滚的旧站版本。

## 2. 已确认的产品定位

采用“个人品牌主导、公司能力承接”的一体化模式。

核心叙事：

> Jeason 把真实业务问题推进成可以运行、检查和持续迭代的工具与系统；倍数引擎是承接企业合作、软件研发和 AI 落地服务的正式主体。

网站不能表现为两个品牌争夺首页，也不能退化成常见的通用软件外包模板。个人经历、项目证据和公司服务必须形成同一条可信路径。

## 3. 目标受众

### 3.1 企业合作方

希望了解团队能否提供软件研发、AI 应用落地、GEO、自动化工作流和部署服务。

### 3.2 潜在客户与合作伙伴

希望快速确认公司主体、联系方式、办公地址、业务范围和实际案例。

### 3.3 技术与内容读者

通过项目页和笔记页了解 Jeason 的产品实践、AI 工作流和研究方法。

### 3.4 搜索与生成式引擎

通过清晰的信息架构、结构化数据、稳定 URL 和事实型内容理解个人、公司、服务与项目之间的关系。

## 6. 服务页设计

服务页不使用“无所不能”的外包叙事。每项服务采用统一结构：

- 适合什么问题
- 可以交付什么
- 如何推进
- 哪些项目可以作为能力证据
- 哪些情况不适合合作

### 6.1 软件与产品研发

覆盖 Web、桌面端、浏览器扩展、后端接口和内部工具。强调从需求到可运行产品，而不是只列技术栈。

### 6.2 企业 AI 应用与落地

覆盖知识工作流、客服智能体、内容处理、数据提取和人机协同。避免承诺不可验证的“全面智能化”。

### 6.3 GEO 与 AI 搜索可见度

覆盖事实资产梳理、网站信息架构、结构化内容、可见度诊断和持续监控。与传统 SEO 的边界必须表达清楚。

### 6.4 自动化与数字化工作流

覆盖重复流程、数据搬运、内容归档、报告生成和跨工具连接。

### 6.5 云端部署与基础设施

覆盖静态站点、API 服务、容器化部署、基础监控和日常运维。只陈述实际可交付能力，不继续使用旧站“高可用、7×24”等无条件承诺。

## 17. 部署与替换方案

### 17.1 本地阶段

1. 在功能分支完成页面与内容改造。
2. 运行 Astro 类型检查和生产构建。
3. 启动本地预览，验证桌面和移动端。
4. 检查中英文、主题、导航、联系链接、结构化数据、Sitemap 和验证文件。

### 17.2 服务器预发布

1. 生成新的版本化 release。
2. 上传到 `/var/www/astro-site/releases/<release>`。
3. 将 `/var/www/astro-site/current` 指向新 release。
4. 先通过服务器 IP 或临时 Host 规则验证。
5. 不立即删除旧 `/var/www/html`。

### 17.3 正式替换

修改 `beishuyinqing.cn` 的 Nginx HTTPS server：

- `root` 从 `/var/www/html` 切换到 `/var/www/astro-site/current`
- 保留现有 SSL 证书配置
- 保留 HTTP 到 HTTPS 跳转
- `nginx -t` 成功后平滑 reload

### 17.4 回滚

若正式域名验证失败：

- 立即把 Nginx root 切回 `/var/www/html`
- 或把 `current` 指回前一个已验证 release
- `nginx -t` 后 reload

旧站目录和旧 release 在确认稳定前不得删除。
